- ✅ 데이터 import 포맷 spec
- ✅ Sim ↔ Real 갭 분석 도구 design

## 1. 동기 / 배경

### 1.1 풀어야 할 질문 (Layer 2)

> Q4. "운영 중 어떤 컨텐츠가 인기 있는가? 그에 따라 배치/레이아웃을 바꿔야 하는가?"
> Q5. "시간대별 관람객 추이는? 어떤 행동 유형이 많은가?"
> Q6. "실데이터로 시뮬레이션 모델을 보정할 수 있는가?"

### 1.2 Layer 1 vs Layer 2

| 차원 | Layer 1 (기획) | Layer 2 (운영) |
|------|--------------|---------------|
| 입력 | 가정 (분포, 가중치) | 실측 (트래킹 로그) |
| 사용자 | 큐레이터, 디자이너 | 운영팀, 컨텐츠 매니저 |
| 검증 | 시뮬 자체 | 실측 vs 시뮬 갭 |
| 의사결정 | 시공 전 결정 | 운영 중 조정 |
| 가치 제안 | "어떤게 좋을까" | "어떻게 바꿀까" |

### 1.3 사용자 통찰

> "운영중인 시설에서 - 컨텐츠 소구력, 어느 컨텐츠가 인기가 많은지 그에 따라 배치나 레이아웃을 달리해야하는지?"
> "운영중인 시설에서 - 시간대별 관람객 추이, 관람객 행동 유형 분석"

## 2. 트래킹 데이터 모델

### 2.1 입력 데이터 종류

실제 시설에서 수집 가능한 트래킹 데이터:

| 종류 | 정확도 | 비용 | 일반적 사용 |
|------|------|-----|----------|
| **CCTV + CV** | 높음 (위치 + 행동) | 매우 높음 | 대형 시설, 보안 겸용 |
| **Wi-Fi probe** | 중 (zone 단위) | 낮음 | 일반적, 익명성 |
| **비콘 (BLE)** | 중-높음 (위치) | 중 | 앱 사용 시설 |
| **티켓 시스템** | 입장/퇴장만 | 매우 낮음 | 모든 시설 기본 |
| **인터랙티브 미디어 자체 로그** | 미디어 단위 정확 | 없음 | 자동 수집 |

VELA 는 위 모든 종류를 **공통 포맷** 으로 추상화.

### 2.2 공통 입력 포맷

```ts
// docs/specs/data-formats/tracking-event-v1.json (참고용 spec)

interface TrackingEvent {
  facilityId: string;
  timestamp: number;       // unix ms
  source: 'cctv' | 'wifi' | 'ble' | 'ticket' | 'media-log';
  
  // 익명 방문객 식별 (세션 단위)
  visitorSessionId: string;  // 일별/방문별 anonymous ID
  
  // 이벤트 타입
  eventType: 'arrival' | 'admission' | 'zone-enter' | 'zone-exit' 
           | 'media-engage-start' | 'media-engage-end' | 'departure';
  
  // 위치 정보 (있을 때만)
  zoneId?: string;
  mediaId?: string;
  position?: { x: number; y: number };  // 시설 좌표계 (m)
  
  // 추가 메타
  metadata?: Record<string, any>;
}
```

지원 import 포맷:
- JSONL (한 줄 한 이벤트)
- CSV (스키마 매핑)
- Parquet (대용량)
- 실시간 스트림 (Phase 4 후반, WebSocket / Server-Sent Events)

### 2.3 데이터 품질 등급

| 등급 | 정의 | 사용 가능 분석 |
|------|-----|------------|
| **A (full)** | 위치 + 행동 + 미디어 engagement 모두 | 모든 KPI, 캘리브레이션 가능 |
| **B (zone-level)** | zone 단위 진입/이탈 + 미디어 로그 | 동선 분석, 미디어 KPI, 부분 캘리브레이션 |
| **C (entry-only)** | 입장/퇴장만 | 시간대 추이, 처리량, 만족도 추정 X |

VELA 는 등급 자동 감지 + 가능한 분석만 활성화.

## 3. 핵심 분석 기능

### 3.1 컨텐츠 소구력 분석 (Q4)

**입력**: 미디어별 engagement 로그 (등급 B 이상)

**출력**:
- 미디어별 도달률, 평균 체류, 의미있는 완주율 (Phase 3B 의 KPI 와 동일 정의)
- 시간대별 인기도 변화
- "낮은 소구력" 식별 (도달률 < 30% 또는 평균 체류 < minWatchMs × 0.5)
- 재배치 시뮬 추천:
  - 후보 1: 위치 X → 위치 Y (입구 가까이) → 예상 도달률 +N%
  - 후보 2: 컨텐츠 길이 5분 → 2분 → 예상 완주율 +N%
  - 후보 3: PASSIVE → STAGED 전환 → 예상 의미있는 완주 +N%

**구현 design**:
- Phase 3B 의 `DigitalMediaAnalyzer` 재사용 (입력 데이터만 다름)
- 재배치 시뮬은 Phase 3A/3B 의 Variant 시스템 활용
- "추천" 산출은 grid search (위치 N 후보 × 길이 M 후보 = N×M 시뮬)

### 3.2 시간대별 추이 + 행동 유형 (Q5)

**입력**: 시간 단위 입장/체류/이탈 로그 (등급 C 이상)

**출력**:
- 시간대별 입장/이탈 그래프
- 평균 체류 시간 분포
- 관람 행동 유형 자동 분류 (Phase 0 의 quick / explorer / immersive 활용)
  - 알고리즘: 체류 시간 + 본 미디어 수 + skip 률 → 클러스터링
- "주말 오후 패턴" / "평일 오전 패턴" 등 자동 프리셋 생성

**구현 design**:
- 신규 모듈: `src/analytics/realdata/PatternExtractor.ts`
- 출력은 시뮬레이션의 `VisitorDistribution` 호환 → 자동 시나리오 생성

### 3.3 Sim ↔ Real 갭 분석 (Q6)

**핵심 질문**: 시뮬 모델이 현실을 잘 예측하는가? 어디서 어긋나는가?

**입력**: 같은 시설의 시뮬 결과 + 실측 데이터

**출력**:
```
                       시뮬 예측    실측        갭        평가
─────────────────────────────────────────────────────────────
입장 처리량/h         180          168         -7%       ✓ 우수
평균 체류 시간         52 min       48 min      -8%       ✓ 우수
미디어 평균 도달률     78%          65%         -17%      ⚠️ 큼
스킵률                22%          35%         +59%      ⚠️ 매우 큼
조기 이탈률            12%          28%         +133%     ❌ 모델 부정확
─────────────────────────────────────────────────────────────
                                                
원인 추정:
  · 스킵률 갭 → engagement 시간 분포가 실제보다 길게 가정됨
    추천: VISITOR_PATIENCE 값 조정 (default 0.6 → 0.45)
  · 조기 이탈 갭 → 피로도 누적이 실제보다 느림
    추천: FATIGUE_RATES 조정 (general 0.00005 → 0.00008)
```

**구현 design**:
- 신규 모듈: `src/analytics/calibration/GapAnalyzer.ts`
- 핵심 KPI 자동 비교 (시뮬 vs 실측)
- 갭 > 임계 시 파라미터 조정 후보 제시
- 자동 캘리브레이션 (Phase 4 후반):
  - 베이지안 최적화 또는 grid search
  - 목표: 핵심 KPI 5종의 평균 갭 < 10%

### 3.4 운영 의사결정 도구 (종합)

**워크플로우**:
1. 트래킹 데이터 import → 실측 KPI 산출
2. 같은 시설의 시뮬 (현재 모델) 실행 → 시뮬 KPI
3. Sim ↔ Real 갭 분석 → 모델 캘리브레이션
4. 캘리브레이션된 모델로 "조정안" 시뮬 (Phase 1-3 활용)
5. 조정안 추천 + 예상 효과

## 5. 데이터 보안 / 익명성

### 5.1 원칙

- **개인 식별 정보 (PII) 절대 보유 X**
- visitorSessionId 는 일별 익명 토큰
- 위치 데이터는 시설 내부 좌표계만 (외부 공간/주소 X)
- 데이터 보관 기간 명시 (default 90 일)

### 5.2 GDPR / 한국 개인정보보호법 호환

- 데이터 처리 동의 절차 (시설 측 책임)
- 사용자 요청 시 세션 데이터 삭제 가능
- 데이터 익명화 검증 (k-익명성 등) — 시설 측 사전 처리 가정

### 5.3 데이터 처리 위치

- 시설 on-premise 처리 권장 (클라우드 업로드 X)
- VELA 는 로컬 데이터 import 만 지원 (Phase 4 초기)
- 클라우드 동기화는 명시적 동의 + 옵션 (Phase 4 후반)

## 6. 구현 단계 (미래 분리)

본 문서가 구현 phase 로 활성화될 때 분해:

### Phase 4.1 — 데이터 import + 기본 분석 (3-4 주)
- TrackingEvent 포맷 + import (JSONL / CSV)
- 데이터 품질 등급 자동 감지
- 기본 KPI 추출 (실측 KPI)
- 운영 대시보드 MVP

### Phase 4.2 — Sim ↔ Real 갭 분석 (2-3 주)
- GapAnalyzer 모듈
- KPI 비교 + 시각화
- 파라미터 조정 후보 추천

### Phase 4.3 — 캘리브레이션 자동화 (3-4 주)
- 베이지안 최적화 또는 grid search
- 캘리브레이션 결과 검증
- 캘리브레이션 모델 저장/관리

### Phase 4.4 — 재배치 추천 (2-3 주)
- 재배치 후보 자동 생성
- 효과 시뮬레이션 (Phase 3A/3B 재사용)
- 추천 워크플로우 UI

### Phase 4.5 — 실시간 스트림 + 클라우드 (4-6 주, 선택)
- WebSocket / SSE 스트림
- 실시간 대시보드
- 클라우드 동기화

총 구현 기간: **약 3-4 개월** (Phase 4.5 제외 시 2.5 개월)

## 8. 비목표 (Non-Goals)

본 design doc 에서 **하지 않는 것**:

- ❌ 트래킹 시스템 자체 개발 (CCTV, Wi-Fi, 비콘 인프라)
- ❌ 영상 처리 / CV 모델 개발 (외부 시스템 결과만 import)
- ❌ 일반 BI 대시보드 (Tableau / Looker 대체)

## 부록 B — Layer 3 (도메인 확장) 연결

Phase 4 의 트래킹 데이터 모델 + 캘리브레이션 워크플로우는 **도메인 확장 시 그대로 재사용**:
- 리테일 매장: 미디어 → 매대, KPI → 통과율/구매율
- 병원 검진센터: zone → 검사실, 미디어 → 검사 슬롯, KPI → 대기시간/처리량

같은 트래킹 추상 + 같은 캘리브레이션 = 도메인 확장 시 별도 인프라 불필요.
